2-1. 서비스 현황 (준비 정도)

AI기반 마이디지털정보 오남용 알림 서비스는 사용자가 인지하기 전에 발생하는 개인정보 및 사진·동영상의 노출·유통을 다채널에서 탐지하고, 알림과 조치 착수로 연결하는 구독형 보호 서비스로 정의한다. 현재 준비도는 “정량 KPI 결과값”이 아니라, KPI의 측정 설계가 문서화되어 있고 산출물(로그·리포트·정책·명세)로 재현 가능한지, 그리고 검증 상태가 완료/부분/미착수로 구분되는지에 기반해 관리한다. 또한 준비 과정에서 드러난 오탐, 프라이버시, 운영 리스크를 각각 어떤 설계와 절차로 통제하는지까지 포함해 “상용화 전 검증 체계”로 정리해 두었다.

가. 개발 이력/추진 경과

서비스 기획 착수 이후, 개인이 스스로 노출을 찾아야 하는 구조가 왜 반복되는지에 대한 원인 분해부터 진행했고, 이를 “탐지 이전의 분산”과 “조치 단계의 복잡성”으로 나누어 요구사항으로 전환했다. 시장조사 및 경쟁분석은 특정 기업 비교가 아니라 대안 유형 단위로 수행했으며, (1) 수동 검색형 대안은 지속적 모니터링과 증거 축적이 사용자 노력에 의존하고, (2) 단일 채널 모니터링형 대안은 재게시·재가공이 발생하는 경로를 요구사항으로 커버하기 어렵고, (3) 조치 대행 단독형 대안은 탐지 근거와 재발 감시가 분리되어 운영 일관성이 깨질 수 있다는 한계를 요구사항으로 환원했다. 이 요구사항을 바탕으로 AI 모델 PoC를 먼저 완료하여 “탐지 후보 생성과 근거 제시”의 가능 범위를 확인했고, 이후 탐지-정규화-중복제어-알림-증거보존-조치착수로 이어지는 사용자 흐름을 재현하는 MVP 개발에 착수했다. 추진 과정의 산출물은 모델 학습·평가 리포트, 채널별 수집·정규화 명세, 이벤트 스키마 및 감사로그 설계, 조치 플레북 초안 등으로 정리되어 있으며, 각 산출물은 동일 입력 조건에서 동일 판단을 재현하도록 만드는 기준선으로 사용한다.

나. 현재 개발 단계(TRL 기준) 및 주요 기술 검증

현재 개발 단계는 TRL 프레임에서 “관련 환경에서의 검증”에 해당하는 위치로 정의하며, 이는 실사용 전 단계에서 관련 조건을 모사한 환경에서 통합 동작을 검증하는 수준을 의미한다.[R1] 해당 TRL 매핑의 근거는 첫째, 환경 측면에서 공개 웹·커뮤니티·테스트 계정 기반 채널과 다크웹 샘플을 포함한 제한된 조건에서 입력 수집과 탐지 후보 생성을 반복 재현하고 있다는 점, 둘째, 통합 측면에서 얼굴 탐지 AI, 개인정보 패턴 매칭, 다크웹 수집/크롤링, 이벤트 정규화, 중복 알림 제어, 증거보존 모듈을 하나의 파이프라인으로 결합해 단일 사건 단위로 처리할 수 있다는 점, 셋째, 운영 측면에서 재시도, 예외 처리, 증거 스냅샷 보존, 오탐 되돌림 흐름을 사용자 알림과 조치 착수에 연결하는 운영 시나리오를 명세와 로그로 남기고 있다는 점이다.[R1] 다음 게이트(Go/No-Go)는 “재시도·예외 상황에서 동일 사건이 중복 생성되지 않고, 근거가 포함된 알림이 생성되며, 오탐 판정 시 되돌림과 증거 폐기/보관 정책이 동일 절차로 수행되는지”를 통과 조건으로 고정하고, 통과 여부는 로그·리포트·정책 문서로 확인되도록 설계했다.

아래 표는 준비도를 정량 값이 아닌 “측정 설계와 증빙 산출물의 존재, 그리고 현재 검증 상태”로 객관화하기 위한 내부 KPI 체계이다.

얼굴 탐지 후보 생성 재현성 동일 입력에서 동일 후보가 생성되는지 동일 입력 반복 실행 비교 내부 로그/테스트 기반, 측정 창: 3주, 표본: 내부 테스터 제공 이미지·영상 샘플, 증빙: 로그/리포트 모델 실행 로그, 후보 생성 리포트, 모델 카드 초안 부분 조명·각도 변형 케이스 추가, 오탐 후보 라벨링 체계 정리
개인정보 패턴 매칭 근거 제시 탐지 결과에 근거(패턴/맥락)가 함께 제공되는지 탐지 결과 레코드의 근거 필드 점검 내부 로그/테스트 기반, 측정 창: 2주, 표본: 이메일·전화·계정 식별자 형태의 테스트 문자열, 증빙: 로그/명세 패턴 룰 명세, 탐지 결과 로그, 근거 필드 정의서 완료 민감정보 과노출 방지 마스킹 규칙 강화, 근거 표시 UI 가이드 반영

증거보존 및 체인 관리 알림 근거 스냅샷과 메타데이터가 추적 가능하게 남는지 증거 저장/조회 흐름 점검 내부 로그/테스트 기반, 측정 창: 2주, 표본: 채널별 탐지 사건 샘플, 증빙: 로그/정책 증거 스냅샷 저장 로그, 보관 정책 초안, 조회 절차 명세 부분 보관 기간·폐기 트리거 정책 확정, 접근권한 감사로그 연결
오탐 되돌림 처리 절차 오탐 판정 시 알림·증거·조치 상태가 일관되게 되돌아가는지 상태전이 테스트 케이스 실행 내부 로그/테스트 기반, 측정 창: 3주, 표본: 내부 테스터 오탐 판정 시나리오, 증빙: 로그/플레이북 상태전이 명세, 되돌림 테스트 리포트, 조치 플레이북 부분 사용자 설명 문구 정교화, 되돌림 후 재학습/룰 개선 티켓 흐름 확정
감사로그 및 접근통제 점검 접근·변경 행위가 감사로그로 남고 통제가 적용되는지 권한별 접근 시나리오 테스트 내부 로그/테스트 기반, 측정 창: 2주, 표본: 운영자·사용자 역할 시나리오, 증빙: 로그/정책 접근통제 정책, 감사로그 스키마, 점검 리포트 완료 운영 점검 주기 정의, 이상 접근 탐지 규칙 정리

표의 해석은, 개인정보 패턴 매칭의 근거 제시와 감사로그/접근통제는 기준선이 정리되어 완료 상태로 관리되는 반면, 채널 수집·정규화·얼굴 탐지·중복 제어·증거보존·오탐 되돌림은 예외 케이스와 운영 절차를 추가로 정리해야 하는 부분 상태로 남아 있다는 점이다.

다. 파일럿/테스트 결과 및 목표시장 반응

파일럿/테스트는 외부 도입이 아니라 내부 테스터 기반으로 진행했으며, 반응은 “무엇이 불명확했는지”와 “어떤 증거가 필요했는지” 중심으로 기록했다. 첫째, 탐지 알림을 받았을 때 사용자는 결과 자체보다 “왜 이 판단이 나왔는지”를 확인할 근거를 요구했고, 이에 따라 증거 스냅샷과 근거 필드(패턴·맥락·발견 경로)를 알림 화면에 연결하도록 설계를 보완했다. 둘째, 삭제 요청 등 조치 착수 단계에서는 사용자가 직접 수행하기 어렵다는 피드백이 반복되었고, 상태 추적(요청 생성-접수-처리-재시도)과 예외 처리(거절, 재게시, 링크 변경)가 동일 절차로 관리되어야 한다는 요구 1순위 요구사항으로 정리되었다. 셋째, 프라이버시 관점에서 알림에 포함되는 정보의 노출 범위가 과도해지지 않도록 마스킹과 최소 표시 원칙이 필요하다는 관찰이 있었고, 이는 오탐과 운영 리스크와 함께 “되돌림 가능 설계”로 연결되어 정책·명세·플레이북의 수정 티켓으로 반영했다. 개선 항목은 알림 근거 제시, 중복 억제, 오탐 되돌림, 증거보존/폐기 정책 순으로 우선순위를 두고 있으며, 각 항목은 테스트 로그와 리포트로 재현 가능한 형태로 남기도록 운영했다.

라. 현재까지 주요 성과 및 다음 단계

현재까지의 주요 성과는 (1) 얼굴 탐지 AI와 개인정보 패턴 매칭, 다크웹 수집/크롤링을 포함한 탐지 파이프라인의 산출물화(명세·로그·리포트로 재현 가능 상태)와 (2) 개인정보보호법 준수 아키텍처 설계의 문서화로 요약된다.[R3] 준수 설계는 선언이 아니라 개인정보 최소 수집, 탐지 근거의 마스킹, 원본과 파생데이터의 분리 보관, 역할 기반 접근통제, 감사로그, 보관·폐기 정책을 데이터 흐름도와 정책 문서로 고정해 검증 가능한 형태로 관리하는 것을 기준으로 한다.[R3] 대외 성과는 한국인터넷진흥원 지원사업 공고를 기준으로 신청 준비를 진행하고, 요구 산출물(서비스 개요, 기술 구성, 개인정보 보호 설계, 검증 계획, 운영 계획)의 제출 형태를 정렬해 접수 가능한 문서 묶음으로 구성한 수준까지로 한정한다.[R2]

대안 유형 대비 결정적 우위는 다음 3가지로 고정한다. 첫째, 탐지 결과에 근거를 결합하고(명세·리포트) 오탐 되돌림과 증거보존 절차(플레이북)를 함께 설계해 운영 리스크를 줄이도록 준비했다. 둘째, 내부 테스터 피드백에서 확인된 “근거 제시”와 “조치 상태 추적” 요구를 알림 UI와 상태전이 명세에 반영하여, 사용자 관점에서 필요한 통제 기능을 산출물로 고정했다. 셋째, 개인정보 최소 수집·접근통제·감사로그·보관 정책을 아키텍처 수준에서 문서화해 프라이버시 리스크를 사후 대응이 아니라 설계 단계에서 통제하도록 구성했다.[R3]

다음 단계는 MVP 고도화와 실사용 조건에 가까운 검증 범위 확대이며, 목표는 수치가 아니라 “통과 조건과 증거 산출물”로 정의한다. 구체적으로는 (1) 채널별 예외 입력에서의 재시도·정규화 일관성, (2) 동일 사건 중복 억제와 사용자 설정 기반 알림 제어, (3) 증거보존/폐기 정책과 접근통제의 일관 적용, (4) 오탐 되돌림 이후 개선 티켓이 운영 흐름에 연결되는지를 통과 조건으로 두고, 각 조건은 로그·리포트·정책·명세로 제출 가능한 형태로 축적한다. 이 과정을 통해 “탐지-근거-통제-조치”가 하나의 운영 단위로 재현 가능하다는 것을 단계적으로 입증하는 방향으로 준비도를 갱신한다.
