2-1. 서비스 현황 (준비 정도)

AI기반 마이디지털정보 오남용 알림 서비스는 사용자가 인지하지 못한 개인정보와 사진·동영상의 노출 및 재유통을 탐지해 알림과 조치 착수로 연결하는 구독형 보호 서비스로 정의한다. 개인이 직접 검색과 신고를 반복하는 방식은 채널 분산, 재게시 속도, 증거 확보의 번거로움으로 인해 일관된 대응을 유지하기 어렵다. 따라서 현재 준비의 초점은 탐지 자체의 성능 주장보다도, 멀티채널 수집과 정규화, 탐지 결과의 증거화, 사용자 알림과 조치 흐름을 하나의 운영 가능한 체계로 묶는 데 있다.

가. 개발 이력/추진 경과
서비스 기획 착수 이후, 목표 고객이 실제로 겪는 불편을 “노출 인지 지연”과 “조치 착수의 부담”으로 재정의하고, 이를 해결하기 위한 탐지-알림-조치 흐름을 단계적으로 설계했다. 시장조사 및 경쟁분석은 특정 기업의 기능 보유 여부를 단정하는 방식이 아니라, 대안 유형을 수동 검색형, 키워드 알림형, 사후 삭제 요청 중심형으로 분류하고 각각의 한계가 어디에서 발생하는지 요구사항으로 환원하는 방식으로 정리했다. 이후 얼굴 기반 이미지 식별, 텍스트 기반 개인정보 패턴 매칭, 폐쇄 커뮤니티 성격의 유통 채널을 포함하는 수집·분석 흐름에 대한 PoC를 수행해, “탐지 이벤트”를 하나의 표준 포맷으로 생성하고 추적하는 내부 산출물을 확보했다. 현재는 이 산출물을 기반으로 MVP 개발을 착수하여, 수집 모듈, 정규화 모듈, 탐지 모듈, 알림 모듈, 증거 패키징 모듈을 단일 파이프라인으로 연결하는 구현 단계를 진행하고 있다.

아래 표는 준비도를 정량 결과값이 아니라, 정량 KPI의 측정 설계와 검증 산출물의 존재, 그리고 현재 검증 상태로 객관화하기 위해 정리한 것이다. 표에서 완료로 표시된 항목은 로그·리포트·정책·명세 형태의 증거가 확보된 상태이며, 부분 또는 미착수로 표시된 항목은 운영 리스크를 줄이기 위한 다음 조치가 남아 있음을 의미한다.

조치 착수 워크플로 정렬 탐지 이후 삭제 요청 등 조치가 표준 절차로 이어지는 수준 조치 유형별 단계, 증빙, 사용자 확인 흐름 점검 조치 워크플로 명세, 템플릿 문서, 처리 상태 로그 미착수 외부 접수 채널별 요구 증빙 정리, 책임 범위와 안내 문구 정합화

나. 현재 개발 단계(TRL 기준) 및 핵심 기술 검증
현재 개발 단계는 TRL 프레임에서 말하는 관련 환경에서의 기술 검증 단계에 해당하도록 목표를 두고, “현실적인 채널 조건에서 통합 파이프라인이 동작하는지”를 기준으로 준비 상태를 정리하고 있다. [R1] TRL 매핑 근거의 첫 요소인 환경은 공개 웹과 커뮤니티성 채널에서 수집되는 텍스트·이미지·동영상 단서를 대상으로, 수집 정책과 접근 제약을 반영한 입력 조건에서 검증을 수행하는 형태로 정의했다. TRL 매핑 근거의 둘째 요소인 통합은 수집, 정규화, 얼굴 탐지 기반 이미지 단서 생성, 개인정보 패턴 매칭, 탐지 이벤트 표준화, 알림 생성, 증거 패키징을 단일 흐름으로 결합해, 모듈 단위 성능 설명이 아니라 end-to-end 산출물이 생성되도록 구성한 점에 있다. TRL 매핑 근거의 셋째 요소인 운영은 재시도, 정규화 규칙, 증거보존, 예외처리의 기본 정책을 명세와 로그로 남기는 수준까지는 진행했으나, 조치 착수의 외부 연계와 운영 권한 분리 체계는 추가 설계가 남아 있어 관련 환경 검증 단계로 정의하는 것이 타당하다. [R1]
주요 기술 검증은 얼굴 탐지 기반의 이미지 단서화, 텍스트 기반 개인정보 패턴 매칭, 폐쇄 채널 성격의 유통 경로를 고려한 수집·크롤링의 세 축으로 나누어 진행했고, 각 축은 검증 범위와 조건을 명세로 고정하고 로그·리포트로 재현 가능성을 확보하는 방식으로 관리했다. 수치 기반 성능 주장 대신, 오탐 발생 시 검토와 되돌림이 사용자 흐름 안에서 처리되도록 설계하고, 탐지 근거가 증거 패키지 형태로 남도록 해 운영 리스크를 줄이는 방향으로 검증을 정렬했다. 다음 게이트는 특정 성능 수치가 아니라, 다채널 입력에서 탐지 이벤트가 안정적으로 생성되고, 알림이 근거와 조치 안내를 포함하며, 오탐 처리가 사용자 경험을 훼손하지 않는 절차로 닫히는 상태를 충족하는지로 정의한다.

다. 파일럿/테스트 결과 및 목표시장 반응
내부 테스터 기반의 시험 운영에서는 탐지 사실 자체보다도 “왜 탐지로 판단했는지”와 “어떤 증거가 남는지”가 알림 수용성을 좌우한다는 점이 반복적으로 관찰되었다. 특히 알림 메시지에 근거가 부족하면 사용자가 직접 재검색을 수행하게 되어 서비스 흐름이 끊기므로, 탐지 이벤트에 원문 스냅샷과 핵심 단서가 함께 제공되어야 한다는 요구가 정리되었다. 또한 사용자는 개인정보와 민감 이미지가 서비스 내부로 과도하게 유입되는 상황을 우려하므로, 수집 항목을 최소화하고 저장을 분리하며 접근기록이 남는 구조와 같이 통제 가능한 설계 증거를 요구하는 경향이 확인되었다. 이 과정에서 개선 항목은 알림 속도 자체의 단순 개선보다, 알림이 발생한 맥락을 이해시키는 UI 구성, 오탐 처리의 되돌림 흐름, 조치 착수에 필요한 증빙 패키지의 구성요소 정리에 우선순위를 두고 정리되었다. 테스트 결과는 특정 만족도 수치로 표현하지 않고, 관찰된 불명확 지점과 요구된 통제 기능을 명세와 로그로 환원해 다음 개발 범위에 반영하는 형태로 축적하고 있다.

라. 현재까지 주요 성과 및 다음 단계
현재까지의 주요 성과는 대외 홍보용 지표가 아니라, 상용화 단계에서 문제가 되기 쉬운 오탐, 운영 불일치, 프라이버시 우려를 줄이는 준비 산출물의 축적이라는 관점에서 정리된다. 얼굴 기반 이미지 단서화, 개인정보 패턴 매칭, 수집·크롤링의 주요 모듈을 PoC에서 MVP 파이프라인으로 연결해, 탐지 이벤트 표준화와 증거 패키징이 함께 생성되는 구조를 구현 중이며, 이는 조치 착수 단계의 재현성을 높이기 위한 기반이 된다. 개인정보 보호법 준수를 선언형 문구로 처리하지 않고, 개인정보 최소 수집, 목적 중심 처리, 분리 보관, 접근통제, 감사로그, 보관 정책과 같은 설계 항목을 데이터 흐름도와 정책 초안으로 구체화하여 준수 리스크를 설계로 관리하는 수준까지 정리했다. [R3] 또한 한국인터넷진흥원 지원사업 선정은 외부 검토가 가능한 형태로 산출물을 정렬하고, 검증 계획을 문서화하여 제출·평가를 통과했다는 점에서 준비 정도를 설명하는 대외 성과로 반영한다. [R2]
대안 유형 대비 결정적 우위는 다음 세 가지로 고정한다. 첫째, 멀티채널 수집과 정규화를 전제로 탐지 이벤트와 증거 패키지를 함께 생성하는 구조를 갖추어, 테스터가 요구한 근거 제공을 산출물로 연결했고 분쟁 대응과 삭제 요청 단계의 불확실성을 줄인다. 둘째, 개인정보 최소 수집과 분리 보관, 접근통제와 감사로그를 설계 산출물로 고정하여, 테스터가 요구한 통제 가능성을 제공하고 법 준수 리스크를 설계 단계에서 낮춘다. [R3] 셋째, 탐지-알림-오탐 처리의 폐쇄 루프를 플레이북과 티켓 로그로 관리하는 운영 설계를 포함해, 조치 착수의 일관성을 확보하고 운영 예외로 인한 누락을 줄인다. 다음 단계는 관련 환경에서의 입력 조건을 넓혀 실사용 검증을 진행하되, 알림 속도와 UI 직관성 개선을 증거 패키지의 설명 방식과 함께 묶어 개선하고, 조치 착수 워크플로를 외부 접수 채널의 요구 증빙에 맞춰 정렬하는 순서로 추진한다.
